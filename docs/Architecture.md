# Architecture.md

## 전체 구조

```
CEO (Human)
  │  승인 / 우선순위 / 방향 수정
  ▼
CTO (Claude, 최상위 orchestrator)
  │
  ├── Plot Liaison Agent ── CEO의 자유형 피드백(감/취향)을 팀 지시로 번역,
  │                          팀 결과물을 CEO용 선택지로 요약 (하이어라키가 아니라
  │                          CEO ↔ 팀 사이 상시 채널 — 어느 단계에서든 호출 가능)
  ▼
Creative Director Agent  ── 세계관/톤 일관성 총괄
  │
  ▼
Project Manager Agent    ── 스프린트 진행, 팀 간 작업 분배
  │
  ▼
Arc Team  ── 전체 에피소드/장면 구조 설계 (최초 1회, 이후 팀들의 공통 입력값)
  │
  ├─ Story Team     (플롯 초안)
  ├─ Character Team  (캐릭터 정합성)
  ├─ Emotion Team    (감정선/페이싱)
  ├─ Editing Team    (문장/구조 교정)
  ├─ Reader Team     (가상 독자 피드백 시뮬레이션)
  └─ Lore Team       (세계관 DB 관리, 모순 탐지)
  │
  ▼
Publisher Agent  ── 최종 조립, 포맷팅, 출력
```

## 기술 스택 결정

### 오케스트레이션: LangGraph (ADR-001)

**추천 이유**

- MUSE의 핵심 동작은 "토론 → 비판 → 수정 → 재토론"의 **순환 구조**다.
  단순 파이프라인(A→B→C)이 아니라 **조건부 루프와 분기**가 필수인데,
  LangGraph는 상태그래프(StateGraph) 자체가 이 패턴을 위해 설계되어 있다.
- 각 Agent의 판단 근거를 상태(State)에 그대로 축적할 수 있어 "No Magic" 원칙(로그 필수)을
  프레임워크 차원에서 강제하기 쉽다.
- CrewAI/AutoGen보다 로우레벨이라 처음엔 코드가 조금 더 필요하지만,
  "토론 종료 조건" 같은 MUSE 고유 로직을 세밀하게 제어할 수 있다. 이 프로젝트는
  범용 멀티에이전트 툴이 아니라 **창작 조직 시뮬레이터**이므로 제어권이 더 중요하다고 판단.

### 메모리/Lore 저장: SQLite + Chroma (로컬 시작, ADR-002)

**추천 이유**

- 장르/세계관이 아직 미정 → 스키마를 미리 못 박는 건 위험. 가벼운 로컬 스토리지로
  시작해 Sprint 1~2에서 실제 데이터 형태를 관찰한 뒤 Postgres+pgvector로 이관하는 게 안전.
- 역할 분리:
  - **SQLite**: 캐릭터 시트, 세계관 설정, 타임라인 등 정형 데이터
  - **Chroma (로컬 벡터 DB)**: 과거 챕터/대사/설정에 대한 의미 검색 (캐릭터 기억, 일관성 체크용)
- 마이그레이션 경로: 트래픽/데이터량이 늘면 SQLite→Postgres, Chroma→pgvector 또는
  managed vector DB로 교체. 인터페이스를 처음부터 Repository 패턴으로 감싸서 교체 비용 최소화.

## 데이터 흐름

### 0단계 — 최초 1회: Arc 설계 (신규)

단편은 웹소설과 달리 처음부터 끝이 정해진 구조이므로, 챕터 단위 반복 작업 전에
**Arc Team이 전체 장면 구조를 먼저 한 번 설계**한다.

- 입력: 확정된 세계관, 캐릭터 시트, 장르/분량 방향
- 출력: 에피소드 로드맵 (예: "이 단편은 5개 장면 — ①도입 ②... ⑤결말", 각 장면의 목적/등장인물/복선 태그)
- 이 로드맵은 이후 모든 장면의 Story Team 작업에 공통 입력값으로 들어간다.
- CEO가 방향을 바꾸면(캐릭터 추가, 결말 수정 등) Arc Team이 재실행되어 로드맵을 갱신한다.

### 챕터(장면) 1개 생성 기준 (0단계 이후 장면마다 반복)

1. PM Agent가 Arc 로드맵에서 "이번에 쓸 장면"을 정의하고, **직전 장면 요약 + 최신 캐릭터/세계관
   DB 상태**를 함께 불러온다 (이것이 챕터 간 연속성을 보장하는 핵심 메커니즘 — 매 장면이
   독립적으로 생성되지 않고 반드시 이전 상태를 이어받는다).
2. Story Team이 (Arc 로드맵 + 직전 요약을 참고해) 플롯 초안 3안 생성 → Creative Director가
   1안 선택 (근거 로그 기록)
3. Character/Emotion Team이 선택된 안에 캐릭터 반응/감정 곡선 주석
4. Writer 역할(Story Team 산출물 기반)이 초안 작성
5. Editing Team + Lore Team이 병렬 리뷰 → 모순/문장 문제 발견 시 **비판 라운드** 진입
6. 비판 라운드: 최대 N회 반복, 합의 도달 또는 PM 강제 종료 (Workflow.md 참고)
7. Reader Team이 가상 피드백 생성 → 로그에 축적 (다음 스프린트 우선순위 입력값)
8. Publisher Agent가 장면 최종본 조립
9. **장면 요약을 생성해 캐릭터/세계관 DB에 반영** (다음 장면의 1번 단계 입력값이 됨) —
   이 단계가 빠지면 매 장면이 이전 내용을 모르는 채로 다시 시작된다.

## 로깅 (No Magic 원칙 구현)

모든 Agent 판단은 아래 스키마로 기록:

```json
{
  "agent": "CharacterTeam",
  "decision": "캐릭터 A의 대사를 반말에서 존댓말로 수정",
  "reasoning": "3장에서 확립된 관계 변화(적대→신뢰)를 반영해야 함",
  "confidence": 0.8,
  "timestamp": "..."
}
```

이 로그는 `Meeting/`, `Sprint/` 문서와 별개로 Sqlite `decision_log` 테이블에 축적된다.
