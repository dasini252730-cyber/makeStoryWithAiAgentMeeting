# Workflow.md

## 핵심 루프: 초안 → 비판 → 수정

```
[초안 생성] → [병렬 리뷰: Editing + Character + Lore + Emotion]
      ▲                          │
      │                    문제 발견?
      │                    ┌──yes──┐
      └──── 수정 반영 ◄────┤       │no
                            └───────┘
                                   ▼
                            [PM 승인] → 다음 단계
```

## 토론 종료 조건 (가장 중요 — 오늘 CTO가 지적한 리스크)

Agent 간 비판 루프가 수렴하지 않고 무한 반복될 위험이 있다. 아래 3중 안전장치를 둔다.

1. **최대 라운드 제한**: 비판 라운드는 챕터당 최대 3회. 3회 초과 시 PM Agent가 강제 종결하고
   "미해결 이슈"로 로그에 남긴 뒤 다음 단계 진행 (사람이 나중에 검토 가능).
2. **수렴 판정**: 각 라운드마다 "이전 라운드 대비 변경 diff 크기"를 측정. diff가 임계값 이하로
   작아지면(즉 더 이상 실질적 수정이 없으면) 수렴으로 간주하고 종료.
3. **비용 상한**: 라운드당 토큰/시간 예산 초과 시 PM이 강제 종료. (API 비용 폭주 방지)

이 세 조건 중 하나라도 걸리면 무조건 종료 — "논쟁이 아직 안 끝났다"는 이유로 계속 돌지 않는다.

## 로그 스키마

모든 라운드는 다음을 기록한다 (Architecture.md의 decision_log 테이블 사용):

- 라운드 번호
- 어떤 팀이 어떤 문제를 제기했는가
- 어떤 수정이 반영/기각되었는가와 그 이유
- 종료 사유 (합의 / 라운드 초과 / 비용 초과)

## 회의 종류와 주기

| 회의 | 주기 | 참석 | 목적 |
|---|---|---|---|
| Daily Standup | 매일 (15분) | CEO, CTO, Agent Leads | 어제/오늘/이슈 공유 |
| Design Review | 주 1회 | CEO, CTO | 아키텍처 변경 검토 |
| Story Review | 주 1회 | CEO, Creative Director | 완성 챕터 품질 검토 |
| Prompt Review | 격주 | CTO | 프롬프트 개선 (Level 3 대비) |
| Retrospective | 스프린트 종료 시 | 전체 | 무엇을 배웠는가 |

## Daily Standup 포맷 (CTO → CEO 보고 템플릿)

```
CTO 보고 — {날짜}

어제: {완료한 작업}
오늘: {진행 예정 작업}
이슈: {발견된 문제 / 필요한 CEO 결정}
```
