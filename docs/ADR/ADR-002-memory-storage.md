# ADR-002: 메모리/Lore 저장은 SQLite + Chroma(로컬)로 시작

## 상태
승인됨 (2026-07-07, CEO 위임 → CTO 결정)

## 맥락
장르/세계관이 아직 확정되지 않아 정형 데이터 스키마를 미리 확정하기 어렵다.
동시에 캐릭터/세계관 일관성 체크를 위해 의미 기반 검색(임베딩)도 필요하다.

## 검토한 대안
1. **PostgreSQL + pgvector** — 확장성 좋으나, 스키마 미확정 상태에서 인프라를 먼저 세우는 건
   과잉 설계(premature commitment) 위험
2. **SQLite + Chroma (로컬)** — 설정 없이 즉시 시작 가능, 스키마를 유연하게 바꿔가며 실험 가능
3. **CTO 추천대로 위임됨**

## 결정
정형 데이터는 SQLite, 의미 검색(캐릭터 기억, 모순 탐지)은 로컬 Chroma로 시작.

## 근거
- Sprint 1~2는 "무엇을 저장해야 하는가" 자체를 알아가는 단계 → 무거운 인프라보다
  빠르게 스키마를 바꿀 수 있는 가벼운 도구가 유리
- Repository 패턴으로 데이터 접근 계층을 감싸서, 추후 Postgres/pgvector로 교체 시
  Agent 코드 변경 없이 저장소만 교체 가능하도록 설계

## 트레이드오프
- 동시 접속/대규모 데이터에는 부적합 (SQLite 특성) — 하지만 현 단계는 단일 저작 파이프라인이므로 문제 없음
- 나중에 마이그레이션 비용 발생 — Repository 패턴으로 최소화

## 재검토 조건
- 데이터량이 SQLite 실용 한계(수 GB, 동시쓰기 병목)에 근접하거나
- 여러 저작 파이프라인을 동시 운영해야 하는 시점에 Postgres+pgvector로 전환 검토
